# Diagram C4

Diagramy poniżej opisują runtime architekturę całego systemu marketplace. Skupiają się na rzeczywistych elementach uruchamianych dla aplikacji, a nie na narzędziach developerskich w monorepo.

## C1 — Kontekst systemu

```mermaid
C4Context
  title System Context diagram for Domovia Marketplace

  UpdateLayoutConfig($c4ShapeInRow="2", $c4BoundaryInRow="1")

  Person(buyer, "Kupujący", "Przegląda oferty i wysyła zapytania")
  Person(seller, "Sprzedający/Agent", "Publikuje oferty i zarządza kontem")
  Person(admin, "Administrator platformy", "Nadzoruje operacje marketplace")

  System(domovia, "Domovia Marketplace", "Marketplace nieruchomości z wyszukiwaniem, auth, dashboardem i widokiem mapy")
  System_Ext(maps, "Dostawca danych mapowych", "PMTiles/R2 oraz zasoby Protomaps/OpenStreetMap")
  System_Ext(geocoder, "Usługa geokodowania", "Photon API", "Geokodowanie i sugestie lokalizacji")

  Rel(buyer, domovia, "Korzysta z marketplace")
  Rel(seller, domovia, "Zarządza ofertami")
  Rel(admin, domovia, "Nadzoruje platformę")
  Rel(domovia, maps, "Pobiera dane mapowe", "HTTPS")
  Rel(domovia, geocoder, "Geokoduje lokalizacje i pobiera sugestie", "HTTPS/JSON")
```

Diagram C1 zachowuje rozróżnienie ról biznesowych, bo to one definiują kontekst systemu.

## C2 — Kontenery

```mermaid
C4Container
  title Container diagram for Domovia Marketplace

  UpdateLayoutConfig($c4ShapeInRow="2", $c4BoundaryInRow="1")

  Person(user, "Użytkownik platformy", "Kupujący, sprzedający/agent lub administrator")
  Container_Ext(maps, "Dostawca danych mapowych", "PMTiles/R2, Protomaps/OpenStreetMap", "Źródło tiles i stylów mapy")
  Container_Ext(geocoder, "Usługa geokodowania", "Photon API", "Sugestie lokalizacji i geokodowanie")

  System_Boundary(domovia, "Domovia Marketplace") {
    Container(front, "front", "Astro SSR + React islands, Cloudflare Workers", "Warstwa UI, SSR i geokodowania marketplace")
    Container(api, "server-axum", "Rust, Axum REST API", "API auth i domeny marketplace")
    ContainerDb(db, "Postgres", "PostgreSQL", "Trwałe dane aplikacyjne")
  }

  Rel(user, front, "Korzysta", "HTTPS")
  Rel(front, api, "Wywołuje API", "HTTP/JSON")
  Rel(front, maps, "Pobiera tiles", "HTTPS")
  Rel(front, geocoder, "Pobiera sugestie i geokoduje lokalizacje", "HTTP/JSON")
  Rel(api, db, "Czyta i zapisuje", "SQL")

  UpdateRelStyle(user, front, $offsetY="-20")
  UpdateRelStyle(front, api, $offsetY="20")
  UpdateRelStyle(front, maps, $offsetX="30", $offsetY="-10")
  UpdateRelStyle(front, geocoder, $offsetX="-30", $offsetY="-10")
  UpdateRelStyle(api, db, $offsetY="20")
```

W C2 role użytkowników są celowo zagregowane do jednego aktora, bo wszystkie wchodzą do systemu przez ten sam kontener `front`.

Najważniejsze odpowiedzialności kontenerów:

- `front`: strony publiczne, logowanie/rejestracja, dashboard, widok mapy ofert, endpoint `/api/geocode`, autocomplete lokalizacji oraz SSR-owe geokodowanie wejścia dla `/listing`.
- `server-axum`: endpointy `/auth/*` i `/api/v1/*`; moduły `auth`, `accounts`, `geo`, `properties`, `listings`, `engagement`.
- `Postgres`: użytkownicy, sesje `auth_session`, nieruchomości, listingi, wishlisty i wiadomości.

## Poza zakresem

- `Turborepo`, GitHub Actions i `compose.yaml` z Autobase są narzędziami developerskimi i nie są częścią runtime architektury systemu.
- Lokalny kontener Photon uruchamiany przez `compose.yaml` jest tylko developerskim provisioningiem zależności geokodowania; diagram modeluje samą usługę geokodowania jako dependency runtime `front`.
- Diagram nie modeluje osobnego BFF, Redis, kolejek ani dodatkowej bazy danych, bo takich elementów nie ma w aktualnym kodzie.
